Responsive Design: adaptarse sin perder lo que eres

Aprenderás a construir páginas que se adaptan a cualquier tamaño de pantalla usando media queries, unidades relativas (px, %, em, rem, vh, vw) y la filosofía mobile-first. Y a través de eso, a reflexionar sobre algo que los adolescentes conocen muy bien: cómo adaptarse a distintos contextos sin perder lo que uno es.

🎬
Para empezar · Tres situaciones reales

Adaptarse sin dejar de ser quien eres

Lee estas tres situaciones antes de abrir el editor.

📱Con compañeros · La misma historia en distintas pantallas

Gabriel construyó su página web personal. Se veía increíble en su computador. Cuando se la mostró a su mamá en el celular, el texto era microscópico, las imágenes se salían de la pantalla y había que desplazarse horizontalmente para leer. La misma página, dos experiencias completamente diferentes. Lo que funciona en una pantalla de 1400px no funciona en una de 390px. Hay que diseñar para ambos — o mejor, empezar por el más pequeño.

💭 ¿Cómo cambias tú según el "tamaño de pantalla" del contexto — cuando estás con amigos, con la familia, en el colegio?
🌊En casa · Adaptarse o doblarse

Cuando la abuela materna viene a quedarse, la dinámica de la casa cambia. La mamá de Sofía habla más bajito, come más temprano, pone menos música. Sofía también ajusta cosas. Pero hay algo que no cambia: Sofía sigue siendo Sofía. Se adapta a la "pantalla" del contexto sin reprogramarse desde cero. Eso es exactamente diseño responsive — el contenido es el mismo, el layout se ajusta.

💭 ¿Cuándo te adaptas a un contexto sin perder lo que eres, y cuándo sientes que te exigen ser diferente de una manera que te cuesta?
📐En el colegio · Primero lo mínimo, luego lo extra

En el proyecto de grado, la docente les dijo algo que nadie esperaba: "Empiecen por lo mínimo que tiene que funcionar bien. Lo adicional viene después." Primero lo esencial, con los recursos más limitados. Luego, si hay más espacio, más capacidad, más tiempo — se agrega. Mobile-first es exactamente esa filosofía: diseñar primero para el dispositivo más pequeño y con más restricciones, y luego expandir para pantallas mayores.

💭 ¿Hay algún proyecto en tu vida donde deberías aplicar "mobile-first" — empezar por lo esencial en lugar de lo ideal?

La página de Gabriel que no funciona en el celular. Sofía que se adapta sin dejar de ser ella. La docente que pide empezar por lo mínimo esencial. Diseño responsive es adaptación con identidad: el contenido no cambia, el layout se ajusta al contexto. Y mobile-first es la filosofía que lo hace bien: primero lo esencial, luego lo expansivo.

01

Unidades: absolutas y relativas

px · % · em · rem · vh · vw — cada una se mide desde un punto diferente

Explicación

Las unidades en CSS se dividen en absolutas (su valor no cambia: px) y relativas (su valor depende de algo más: el padre, la raíz o el viewport). Las unidades relativas son la base del diseño responsive porque se adaptan automáticamente al contexto.

px
Píxel — absoluta
Siempre el mismo tamaño. No se adapta. Útil para bordes y sombras precisas.
%
Porcentaje — relativa al padre
Se calcula sobre el elemento padre. width: 50% = la mitad del padre.
rem
Root em — relativa a la raíz
Se calcula sobre el font-size del <html> (por defecto 16px). 1rem = 16px. La más recomendada para tipografía y espaciado.
em
Em — relativa al padre
Se calcula sobre el font-size del elemento padre. Útil para padding relativo a la fuente.
vh
Viewport Height
1vh = 1% de la altura de la pantalla. 100vh = pantalla completa en alto.
vw
Viewport Width
1vw = 1% del ancho de la pantalla. Útil para elementos que escalan con el ancho.
unidades.css
/* px: absoluto, siempre igual */ body { font-size: 16px; } /* base fija */ .borde { border: 1px solid #e2e8f0; } /* rem: relativo a la raíz (html font-size) */ /* Si html { font-size: 16px }, entonces: */ h1 { font-size: 2rem; } /* 32px */ h2 { font-size: 1.5rem; } /* 24px */ p { font-size: 1rem; } /* 16px */ .card { padding: 1.5rem; } /* 24px */ /* % : relativo al padre */ .sidebar { width: 30%; } /* 30% del contenedor padre */ .main { width: 70%; } /* vh / vw: relativo a la pantalla */ .hero { min-height: 100vh; } /* toda la altura de pantalla */ .titulo-xl { font-size: 5vw; } /* escala con el ancho de ventana */ .banner { width: 100vw; } /* todo el ancho de pantalla */ /* clamp(): tamaño entre mínimo y máximo */ h1 { font-size: clamp(1.5rem, 4vw, 3rem); } /* mínimo: 1.5rem — preferido: 4vw — máximo: 3rem */

Regla práctica: Usa rem para tipografía y espaciado (respeta la preferencia del usuario), % para anchos de contenedores, px para bordes y sombras, vh/vw para secciones de pantalla completa. Evita usar px para tipografía — no respeta el zoom del usuario.

02

@media: cambiar el estilo según el contexto

La instrucción que dice "si la pantalla es mayor a X, aplica esto"

Explicación

Una media query es una condición en CSS. Dice: "si la pantalla cumple esta condición, aplica estas reglas". La condición más común es el ancho de la pantalla (min-width o max-width). La sintaxis siempre comienza con @media.

media-queries.css
/* ════ SINTAXIS DE UNA MEDIA QUERY ════ */ @media (min-width: 768px) { /* Estilos que aplican SOLO cuando la pantalla ≥ 768px */ .contenedor { grid-template-columns: 1fr 1fr; /* 2 columnas en tablet */ } } @media (min-width: 1024px) { .contenedor { grid-template-columns: 240px 1fr 200px; /* 3 en desktop */ } .nav-movil { display: none; } /* ocultar menú móvil */ .nav-desktop { display: flex; } /* mostrar nav desktop */ } /* ════ BREAKPOINTS ESTÁNDAR (mobile-first) ════ */ /* Base: móvil (sin media query) < 640px */ /* @media (min-width: 640px) sm: tablet pequeña */ /* @media (min-width: 768px) md: tablet */ /* @media (min-width: 1024px) lg: laptop */ /* @media (min-width: 1280px) xl: desktop */ /* También se puede combinar: */ @media (min-width: 640px) and (max-width: 1023px) { /* Solo tabletas */ } /* Preferencia del sistema: modo oscuro */ @media (prefers-color-scheme: dark) { body { background: #0f172a; color: #e2e8f0; } }

03

Mobile-first: diseñar desde lo más pequeño

Primero lo esencial. Luego lo expansivo.

Explicación

Mobile-first significa escribir los estilos base para la pantalla más pequeña (el celular) y usar @media (min-width: ...) para agregar complejidad en pantallas mayores. Lo opuesto —desktop-first— empieza desde lo complejo y usa max-width para reducir. Mobile-first es la práctica estándar hoy porque más del 60% del tráfico web viene de celulares.

✅ Mobile-first (recomendado)
/* Base: 1 columna (móvil) */
.grid { grid-template-columns: 1fr; }

/* Tablet: 2 columnas */
@media (min-width: 768px) {
  .grid { grid-template-columns: 1fr 1fr; }
}

/* Desktop: 3 columnas */
@media (min-width: 1024px) {
  .grid { grid-template-columns: 1fr 1fr 1fr; }
}
❌ Desktop-first (evitar)
/* Base: 3 columnas (desktop) */
.grid { grid-template-columns: 1fr 1fr 1fr; }

/* Tablet: reducir a 2 */
@media (max-width: 1023px) {
  .grid { grid-template-columns: 1fr 1fr; }
}

/* Móvil: 1 columna */
@media (max-width: 767px) {
  .grid { grid-template-columns: 1fr; }
}
mobile-first-completo.css
/* ════ PÁGINA COMPLETA CON MOBILE-FIRST ════ */ /* BASE — móvil primero (sin media query) */ body { font-size: 1rem; padding: 1rem; } .pagina { display: grid; grid-template-areas: "header" "nav" "main" "aside" "footer"; /* en móvil: todo en una columna */ gap: 1rem; } .nav { display: flex; flex-direction: column; gap: 0.5rem; } /* TABLET: ≥ 768px */ @media (min-width: 768px) { .pagina { grid-template-columns: 1fr 1fr; grid-template-areas: "header header" "nav nav " "main aside " "footer footer"; } .nav { flex-direction: row; } /* nav horizontal */ } /* DESKTOP: ≥ 1024px */ @media (min-width: 1024px) { body { padding: 2rem; font-size: 1.0625rem; } .pagina { grid-template-columns: 240px 1fr 200px; grid-template-areas: "header header header" "nav nav nav " "main main aside " "footer footer footer"; } }
El viewport meta tag es indispensable: Sin <meta name="viewport" content="width=device-width, initial-scale=1.0"> en el <head>, los celulares muestran la página como si fuera una pantalla de escritorio — pequeñísima. Ese tag le dice al navegador móvil que use el ancho real del dispositivo. Ya lo aprendiste en la Guía 09 — ahora ves por qué es tan crítico.
No abuses de las media queries: Si necesitas más de 4 o 5 breakpoints, probablemente el problema está en el diseño base. Flexbox con flex-wrap: wrap y Grid con repeat(auto-fit, minmax()) hacen mucho responsive sin necesitar media queries.
A1

Mi proyecto personal, responsive: que funcione en cualquier pantalla

Individual · 25 minutos · Autoconciencia — adaptar sin perder lo que eres
⏱ 25 min 👤 Individual 💻 CSS externo 🔍 SEL: Autoconciencia
📱Como Gabriel — que funcione en el celular de la mamá también

Gabriel hizo la página para su pantalla. Esta actividad hace que funcione para todas. Pero más allá de la técnica: adaptarse a distintos contextos sin cambiar el contenido — el mensaje, la identidad, lo que quieres decir — es lo que hace que una página (y una persona) sea genuinamente responsiva.

💭 ¿Cómo se ve tu proyecto en el celular ahora mismo? ¿Funciona o se rompe?
  • 1

    Revisa el viewport meta tag en todas tus páginas

    Abre el inspector del navegador (F12 → ícono de dispositivo móvil). ¿Cómo se ve tu proyecto en 390px de ancho? Si se ve roto, el primer paso es verificar el meta viewport.

  • 2

    Convierte todos los tamaños de fuente a rem

    En tu estilos.css, reemplaza todos los font-size en px por rem. Si usabas 16px, cambia a 1rem. Si usabas 24px, cambia a 1.5rem. Si usabas 32px, cambia a 2rem.

  • 3

    Aplica mobile-first al layout de tu index.html

    En el CSS base (sin media query): el grid en una columna, el nav en columna. En @media (min-width: 768px): el layout de dos columnas. En @media (min-width: 1024px): el layout completo con sidebar.

  • 4

    Aplica al menos un breakpoint por página

    Cada una de tus 4 páginas debe tener al menos una media query que cambie algo relevante entre móvil y desktop. No tienen que ser todas iguales — cada página puede necesitar ajustes diferentes.

  • 5

    Agrega modo oscuro como bonus

    Con @media (prefers-color-scheme: dark), define al menos el color de fondo y de texto para modo oscuro. Es una media query de preferencia del usuario — una adaptación que respeta su contexto.

📋 Tu CSS debe incluir
  • Viewport meta tag verificado en todas las páginas
  • Tipografía en rem, no en px
  • Layout mobile-first con mínimo dos breakpoints (768px y 1024px)
  • Al menos una media query por página del proyecto
  • Bonus: @media (prefers-color-scheme: dark)
  • Comentarios CSS que expliquen qué cambia en cada breakpoint y por qué
🔍 SEL · Autoconciencia

Adaptar el layout sin cambiar el contenido es la definición de identidad estable

El diseño responsive hace algo que también hacemos las personas: cambia cómo se presenta según el contexto, pero el contenido — lo que dice, quién es — permanece igual. La autoconciencia incluye "mantener una identidad estable en distintos contextos". En tu bitácora:

  • → ¿Cómo se ve tu proyecto en el celular versus el computador? ¿El mensaje principal sigue siendo igual de claro?
  • → ¿Hay algo de tu página que debería mostrarse diferente según el contexto pero que aún no hayas ajustado?
  • → ¿En qué contextos de tu vida cambias cómo te presentas pero mantienes quién eres — como el diseño responsive — y en cuáles sientes que el contexto te cambia demasiado?
  • → ¿Elegiste activar el modo oscuro en tu proyecto? ¿Qué significa para ti que una pantalla respete la preferencia visual del usuario?
A2

Auditoría responsive: revisar el proyecto del compañero

Parejas · 20 minutos · Habilidades relacionales — dar y recibir feedback constructivo
⏱ 20 min 👥 Parejas 💻 Inspector del navegador 💬 SEL: Habilidades Relacionales
🔍Ver desde los ojos del otro

Gabriel no vio el problema en su página porque siempre la miró desde su pantalla. A veces necesitamos que alguien más lo vea desde un ángulo diferente. Esta actividad propone hacer lo mismo: revisar el proyecto del otro en distintos tamaños de pantalla y dar feedback concreto, constructivo y respetuoso.

💭 ¿Cómo recibes el feedback sobre tu trabajo? ¿Qué diferencia hay entre feedback que ayuda y feedback que hiere?
  • 1

    Abran el proyecto del otro en el navegador con el inspector (5 min)

    F12 → ícono de responsive (📱). Prueben en 390px (iPhone), 768px (iPad) y 1280px (laptop). Tomen nota de al menos 3 cosas que funcionan bien y 2 que necesitan ajuste.

  • 2

    Escriban el feedback en un archivo HTML (8 min)

    Archivo feedback_responsive.html. Con una sección por breakpoint analizado. Para cada problema encontrado: qué pasa, en qué pantalla, y una sugerencia concreta de CSS para solucionarlo.

  • 3

    Intercambien y conversen sobre el feedback (7 min)

    ¿El feedback fue claro? ¿Hubo algo que sorprendió — positivo o a mejorar? ¿Cómo se sintió recibir la opinión de alguien sobre algo que construiste?

📋 Protocolo de feedback
  • Primero lo que funciona bien — al menos 3 cosas concretas
  • Luego lo que necesita ajuste — con sugerencia específica, no solo crítica
  • Lenguaje técnico + lenguaje de diseño: "en 390px el texto del nav se superpone con el logo — podrías usar flex-direction: column en ese breakpoint"
  • El feedback sobre diseño es sobre el trabajo, no sobre la persona
💬 SEL · Habilidades Relacionales

Dar y recibir feedback constructivo es una habilidad que se practica

Las habilidades relacionales incluyen "comunicarse claramente" y "resolver conflictos constructivamente". El feedback sobre una página web — como el feedback sobre cualquier trabajo — puede darse de forma que construye o de forma que derriba. La diferencia no está en lo que se dice sino en cómo se dice y con qué intención. En tu bitácora:

  • → ¿Cómo se sintió recibir feedback sobre algo que construiste? ¿Qué necesitaste para poder escucharlo?
  • → ¿Hay algo del feedback que recibiste hoy que podrías aplicar no solo al código sino a la forma en que das feedback en otros contextos?
  • → ¿Fue más fácil dar el feedback técnico (el CSS) o el de diseño (las decisiones de presentación)? ¿Por qué?
A3

Una sola página, tres pantallas: el contexto no cambia lo que somos

Grupos de 4 · 20 minutos · Conciencia social — adaptarse sin perder la identidad colectiva
⏱ 20 min 👥 Grupos de 4 💻 Archivo colectivo 🌍 SEL: Conciencia Social
🦋Como Sofía — adaptarse sin reprogramarse

Sofía ajusta su comportamiento cuando viene la abuela, pero sigue siendo Sofía. El grupo va a hacer lo mismo con su página colectiva: el contenido — quiénes son, qué dicen de sí mismos — no cambia. Solo el layout se adapta a cada pantalla.

💭 ¿Hay algo de la identidad del grupo que debería ser constante sin importar el "tamaño de pantalla" del contexto?
  • 1

    Tomen la página del equipo de la Guía 14 o 15 (si existe)

    O creen una nueva página del grupo con al menos 4 tarjetas de integrante y un header colectivo. El contenido no cambia entre versiones.

  • 2

    Implementen los tres estados responsive (10 min)

    Base (móvil): una tarjeta por fila, nav en columna, header centrado. @media (min-width: 640px): dos tarjetas por fila. @media (min-width: 1024px): cuatro tarjetas en una fila, sidebar con info del grupo.

  • 3

    Discutan: ¿qué del grupo es "el contenido que no cambia"? (5 min)

    Independientemente del tamaño de pantalla, ¿qué de su identidad como grupo debe estar siempre visible y claro? ¿Hay algo que se ve mejor en móvil que en desktop, o viceversa?

📋 Producto esperado A3
  • Página del equipo con tres estados responsive bien implementados
  • Comentarios CSS que expliquen qué cambia en cada breakpoint y por qué
  • Un párrafo colectivo en el footer: ¿qué de la identidad del grupo no cambia con la pantalla?
🌍 SEL · Conciencia Social

Los grupos también necesitan adaptarse a distintos contextos sin perder lo que los une

La conciencia social incluye "comprender las influencias de los sistemas sobre el comportamiento". Un grupo escolar se presenta diferente en una exposición que en una reunión informal — pero sus valores, su dinámica, lo que los une, debería ser consistente. El responsive design enseña esa lección técnicamente: el layout cambia, el contenido no. En tu bitácora:

  • → ¿Qué del contenido del grupo se "ve mejor" en pantalla grande (con más espacio para mostrar)? ¿Y qué se ve igual de bien en móvil?
  • → ¿Hay algún contexto real donde el grupo se presenta diferente a lo que realmente es? ¿Qué los lleva a hacerlo?
  • → ¿Qué aprendiste sobre adaptarse sin perder identidad — en CSS y en la vida?

Criterios de valoración · Guía 16

¿Cómo se evaluará tu trabajo?

CriterioBásico (1–2)Satisfactorio (3)Superior (4–5)
Unidades relativas
Técnico		 Toda la tipografía y el espaciado en px. Tipografía en rem. Anchos en % o fr. Al menos vh/vw usado en un elemento. Cada unidad fue elegida por una razón: rem para tipografía, % para anchos, vh para secciones de pantalla completa. clamp() como bonus.
Media queries mobile-first
Técnico		 No hay media queries o usa max-width (desktop-first). Al menos dos breakpoints con min-width que cambian el layout de forma significativa. El proyecto funciona perfectamente en móvil, tablet y desktop. El layout base es de una columna y crece con min-width.
Adaptación coherente
Técnico + SEL		 El layout cambia pero el contenido se ve roto o incompleto en alguna pantalla. El contenido es legible y accesible en las tres pantallas. En cada tamaño de pantalla, lo más importante del contenido se ve primero y con claridad. El layout sirve al mensaje.
Feedback constructivo
SEL · Habilidades Relacionales		 El feedback fue solo crítica sin sugerencias concretas. El feedback tiene al menos 3 aspectos positivos y 2 sugerencias con CSS específico. El feedback combina lenguaje técnico y de diseño, es respetuoso, y se nota que quien lo escribió genuinamente quería que el proyecto mejorara.
Adaptación e identidad
SEL · Autoconciencia		 La reflexión no conecta el responsive design con situaciones personales. Identifica al menos un contexto real donde se adapta sin perder identidad. La reflexión es precisa y honesta: distingue entre adaptarse saludablemente y cambiar bajo presión lo que no debería cambiar.
